2 Introduction
2.1 Brief Overview
The Storage and Communication Subsystem (UFS) for NVL consists of only UFS controller with a MIPI MPHY and supporting
integration logic. Synopsys provides UFS Host controller and MPHY. UFSHC top wrapper integrates Synopsys UFS Host Controller,
MPHY and ancillaries like IOSF2AXI bridge, AXI fabric etc., to enable integration of UFSHC sub-system in Intel SoC. The top wrapper is
referred to as SCS_UFS in this document. The actual RTL top level is called scc_ufs_wrapper_top.
UFS Host controller is Synopsys UFS controller which is compliant to JEDEC UFSHC v4.0. The interface to PHY is Unipro v2.0 and the
PHY is MIPI MPHY v5.0.The UFS subsystem is located in the AXI domain under the IOSF2AXI bridge which converts the IOSF traffic to
and from the internal AXI protocol which the various IPs support.
2.2 Block Diagram
Figure: SCS_UFS Block Diagram (source)
The above block diagram shows the SCS_UFS subsystem. Yellow colored boxes are delivered by Synopsys, AXI fabric is generated from
Synopsys coreAssembler using DW components. Rest of the logic is built by Intel.
Components
The SCS_UFS consists of the following blocks:
1. IOSF2AXI Bridge.
2. Synopsys DWC AXI Fabric.
3. UFSHC Wrapper with
a. Synopsys UFSHC Controller
b. Synopsys MIPI MPHY
c. APB interface for register access and out of band access
d. Clock gating and muxing logic
4. PGA (Phy Globals Adapter)
5. Convergence layer specific to UFS and Misc. logic

2.2.1 IOSF2AXI Bridge
The IOSF2AXI Bridge serves as the gateway of the SCS_UFS into the IOSF domain. It has two main functions:
1. Convert from IOSF to/ from AXI Bus protocol: The bridge converts transactions from IOSF to the IP’s native bus interface of AXI. It has
two IOSF endpoints.
Primary IOSF endpoint that connects to the SoC PSF.
Sideband IOSF endpoint that connects to the sideband topology.
2. Provide full PCI compatibility for AXI native IPs: The bridge contains the UFS IP’s PCI configuration space parameters as the IP is
natively AXI and do not support PCI configuration parameters.
2.2.2 AXI Fabric
The AXI fabric provides the interconnect inside the SCS_UFS subsystem. The AXI fabric connects to the bridge on one side, and Host
controller and convergence register on the other side.
2.2.3 UFS Sub-System
2.2.3.1 UFS Controller
UFS adopts the latest technology from other industrial standards
MIPI UniPro specification for interconnect layer
MIPI M-PHY specification for physical layer
INCTS T10 UFSI standards (SBC, SPC and SAM) for command sets and architecture model
This product conforms to JEDEC UFS Host Controller Interface Specification JESD220F. The UFS host controller provides an
interface method of accessing the UFS hardware capabilities. The UFS Host Controller handles UFS Protocol at transmission level,
packing data, adding cyclic redundancy check (CRC), start/end bit, and checking for transaction format correctness.
2.2.3.2 SNPS MIPI MPHY
For NVL, Synopsys MIPI MPHY is used. The PHY supports Gear 5.
2.2.4 Phy Globals Adapter (PGA)
PGA includes various blocks to support integration of Synopsys MIPI MPHY with SOC as shown in Figure below. It includes various
blocks like SRAM, JTAG port, IOSF-SB for PGA and SRAM loading, Clocking and PMC handshake interface. PGA consists two functional
sub-blocks as described and shown below.

2.2.4.1 AIC Buttress
AIC buttress includes logic required to chassify Phy interface to SOC to simplify and standardize interface and flows for SOC. AIC
buttress includes various standard widgets like SBEP, Fuse Puller, PGCB and CDC. It also houses registers required to support PHY and
PIA.
2.2.4.2 PIA (PHY interface Adapter)
PIA has logic and state machines to sequence and drive the signals required by PHY. It is configurable to support multiple PHYs.
2.2.5 Convergence layer and Misc Logic
Convergence layer includes various registers needed specifically for UFS sub-system and miscellaneous logic.

5.2.3 Multi Root Space Support
5.2.3.1 Configuration Writes
Bridge ignores root space attribute for downstream Configuration accesses targeting the PCI configuration spaces within the Bridge.
Both host and ESE can modify the configuration space.
5.2.3.2 Downstream MMIO transactions from root space 0
All transactions follow the address decode as required for BAR0 or BAR1.
5.2.3.3 Downstream MMIO transaction from root space 1
All transactions are checked against the standard decode rules for BAR0 and BAR1.
For address decode matching BAR0 and MSE and device power state is D0 then the access is passed downstream following
address remap using the top level remap straps.
For address decode not matching BAR0 or if MSE is 0 or device is in D3, then all downstream accesses are passed as is
(subtractive) without address translation.
ESE is the POR with RS0. But should the CSE be used, The associated axi_user is set as 1 indicating a CSE root space access.
If the address matches BAR1 with MSE set as 1, the access is treated as a BAR1 access in the ACPI mode.
5.2.3.4 Upstream Access
The Bridge will use axi_user in order to distinguish between RS0 and RS1. It will update RSx bit according to axi_user. RS0 is the POR for
NVL.
Bridge ignores downstream root space for access coming from SB towards the Bridge.

Mrd (0x0) and Mwr (0x1) are used by the bridge to access the AXI domain based on BAR addresses. The Bridge uses the BAR register
of the PCI configuration space to add the offset from the Sideband access to the BAR field in the PCI configuration space. This
combined value is remapped as on the IOSF primary, because of the fixed address decode nature of the AXI fabric. BAR1 aliasing
cannot be accessed through the Sideband. The Bridge supports both 16-bit and 48-bit addressing for memory accesses on the IOSF
Sideband. It is not possible to issue an access to a offset bigger than the BAR size. i.e. these SB Msg can’t be used to access the AXI
fabric internal registers which are not BAR mapped.
CRRd (0x6) and CRWr (0x7) are used to access the Fabric register space. The Bridge expects the Register Access message on the
IOSF Sideband to be an offset address and must be above the 4 KB region, that is the address must be 0x1000 or above.
CRRd and CRWr are also used to access PGA registers as well as to program FW in PHY SRAM in PGA, through IOSF-SB interface.

6 Functional Details

6.2 Address Map
This section documents the AXI address mapping. The UFS SS is an AXI subsystem within the SoC and as such the address map of its
components is fixed. It is the responsibility of the IOSF2AXI Bridge to convert the IOSF domain address to the AXI domain address.
Each IOSF transaction to the BAR address region is converted to a matching AXI fixed address value.
Since AXI addresses are fixed (No BAR concept in AXI) the UFS SS internal AXI addresses shall be in a reserved address location within
the IOSF address domain. This is to ensure that the OS will never place a device within this address space during enumeration. i.e. no
device BAR address shall ever be equal or in the range of the UFS SS internal AXI address space. UFS PCICFGCTRLx offset is 0x200.
The MMIO address space for UFS 4.0 has been updated to 32 KB to accommodate Inline Encryption and UFS4.0 Multi Circular
Queues.
16KB UFS IP address space is further divided into two regions:
The Device internal registers - a 12KB address space which maps the devices internal registers.
Extended Configuration registers - a 4KB address space which maps all the extended configuration registers that are not integrated
inside the IP.
These include various configuration options for the IP etc. From the Host standpoint the 32KB is a continuous MMIO space of the
device. The AXI fabric routes the transactions to the correct location based on the above separation.
Within the allocated address range, there are various empty addresses which do not contain any registers. The baseline assumption
and constraint is that there will never be accesses to these regions - NOT by the host directly and NOT by configuring the DMA. (This is
because these addresses are NOT part of the OS address map). Any access to these areas will result in the fabric addresses decode
error and an error response to the initiator.
From the Host view point, UFS has a 32KB BAR address space. But only 16 KB is used. UFS can be configured to work in either PCI or
ACPI modes. The configuration is done via a write to the IOSF2AXI private register by a sideband access by BIOS or FW. For NVL, only
PCI mode is supported.
By default all the devices are defined as PCI devices.

6.4.1.6 Fabric Errors
6.4.1.6.1 Default subordinate
A default subordinate is shared between all address layers and is attached to Secondary Port 0, which is not available as an external
port. The default subordinate can respond simultaneously to one read and one write command.
Read and write commands from external managers are routed to the default subordinate when:
An external manager generates a read/write command where the read/write address is not memory mapped.
A manager is trying to access a secure subordinate with an insecure access. (N/A in NVL use case)
An AXI manager generates a read/write command to a subordinate that is not visible to it.
The default subordinate performs the following:
Accepts at most one active read command and one active write command.
For a read command:
Returns DECERR on the rresp bus for each beat of the read burst. The read data is held constant at 0.
Asserts rvalid until the burst is complete. When a read command is accepted by the default subordinate, rvalid is immediately
asserted and remains asserted until the burst completes.
Asserts rlast on the last beat of the transfer.
For a write transfer:
Does not insert WAIT states on a data transfer - wready is tied high.
Ignores write data.
Generates the write response immediately after the write data burst has completed. The bresp signals a DECERR
6.4.1.6.2 Deadlock
It is possible to have a partial or complete deadlock due to events beyond the control of the DW_axi. Examples of possible causes for
deadlock are:
Incorrect power-down of a subordinate or manager
Internal deadlock on a subordinate or manager
This type of deadlock condition can be detected by monitoring the non-completion of outstanding transactions. If any outstanding
transaction does not complete within a predetermined time interval (128 axi clocks), a deadlock notification is issued.
The deadlock notification signals are connected to the VISA network of the UFS subsystem. The deadlock error notification irq is not
connected to the iosf2axi bridge as the support for oob_fabric_error has been removed from iosf2axi bridge (It was present in iosf2ocp
bridge.)

true hardware PCI interface. This implies an AXI IP must have:

Each AXI IP attached to an AXI port on the Bridge is assigned a unique Bus/Device/Function by means of a strap and Bus Number

6.4.2.1.2 ACPI mode of operation
NVL UFS is NOT expected to be configured in ACPI mode. Bridge allows AXI Fabric to appear as ACPI devices on the system. In this
mode of operation, PCI configuration access to the Bridge will return UR status.
A control bit in the private configuration space disables PCI configuration accesses in the Bridge. However the values programmed in
PCI Configuration registers are still valid. Bridge uses these registers (For example BAR, MSE, D3, BME) to determine how to handle
Memory accesses.
This feature can be enabled by setting the PCI_CFG_DIS bit in the Private configuration space. An additional BAR1 register is present in
the PCI config space of the Bridge. Memory accesses to BAR1 region accesses the PCI configuration space. i.e BAR1 is an alias to the
PCI configuration space. Address decode for BAR1 ignores the MSE and D3 state of the device in the PCI configuration space. The use
model for BAR1 is to access PCI configuration space registers when the PCI configuration accesses have been disabled. MMIO writes
are posted whereas PCI configuration accesses are non-posted. A read should always be performed following BAR1 access to ensure
Bridge observes the BAR1 write access. Only 1DW BAR1 downstream accesses are supported.

6.4.2.2.2 MSE (Memory Space Enable)
MSE is defined within the IOSF2AXI Bridge as part of the Type0 PCI configuration space each device has. When disabled no
downstream traffic from the bridge is available.

UFS SS supports systems with VT-d enabled
1. Can’t know that a DMA address is a true Physical address
2. Virtual address may alias to a physical address programed in a BAR.
3. When VT-d enabled all virtual addresses must route upstream
VT-d performs address translation.
1. From: GPA - Guest Physical Address
2. To: HPA - Host Physical Address.
When VT-d is enabled
1. All I/O Devices must assume the address is a GPA
2. GPA must be routed to VT-d engine for translation
3. After translation address can be routed by address
Support for VT-d is achieved by using AXI physical address that are much bigger than any possible physical memory in the system (AXI

6.4.2.2.15 Interrupt Coalescing
Interrupt coalescing is used to reduce the number of interrupts from the UFS to the Host. The idea is to enable the PMC to capture the
interrupt from the host controller before it reaches the host thus reducing the number of interruption to the normal flow or the host.
Many interrupts are informative only, and there are many flows in which these interrupts can be ignored. Since there is no way to know
the reason of the interrupt beforehand, today the host is required to service each one.
Interrupt Coalescing enable the PMC to first service the interrupt at the host controller. Only if this interrupt is relevant to the host will
the interrupt be send to the host. If the interrupt is informative only and can be ignored, the PMC will clear the interrupt status register
and prevent this interrupt from reaching the host.
To enable this flow the interrupt line from UFS host controllers will be routed to the PMC and only then routed to the Host.
If product do no use interrupt-coalescing, the output interrupt lines from the UFS SS should be looped back to the input interrupts into
the bridge inside UFS SS. Loop back is the POR for NVL. Refer to Integration HAS excel for loopback information.

6.4.3.3.5 Controller
The Controller is responsible for managing the UFS protocol between the UFS host controller and the external UFS device. While UFS
requests are sent to the UFS host controller core in a single command, they may trigger a series of UFS packets transferred between the
host and external device. The Controller provides several different functions:
Managing low level UFS protocol between device and host
Managing the response from the device and store the appropriate information in system memory for the driver level
Triggering the DMA engine to fetch/store data
Managing scatter/gather table and set up DMA engine accordingly
Maximize throughput by intelligent pre-fetching, buffering and pre-configuration
Minimize required bus bandwidth by reducing unnecessary bus activities.
6.4.3.3.6 Arbiter
To deliver optimal performance, the host controller’s initiator interface needs to perform transactions on the system bus without adding
additional delays. While the specific implementation of the AXI protocol mainly determines the available bandwidth on the system bus,
the arbiter determines the order in which requests are being executed.
Although not directly protocol related, the order also has an impact on the available bandwidth of the system - especially in systems
with a high latency to access the system memory. In those systems, it is important to hide the latency introduced by the access time
behind a series of similar requests. Those could be a block of read or write requests, or a request stream with alternating types, for
example, read followed by write, followed by read, and so on.
While on one side the arbiter is connected to the initiator interface, on the other side, the interface is connected to five different internal
clients. The arbiter manages based on priority and bandwidth requirements on the access to the system bus. Additionally, there is one
high priority request coming from one of the clients.
The internal clients and the resulting requests are based on the internal architecture and the functionality that the IP needs to perform:
High Priority Access: A special access for task management requests. This request is initiated by the controller unit and is combined
with a special high priority signal. These are only read requests.
Command Interface Port Unit: The Command Interface Port Unit is responsible for fetching new transaction list entries, task
management entries, command headers and Physical Region Descriptors (PRDs). The unit only creates read requests with a burst
size which depends on the size of the requested information.
Save Command Status Unit: Saves result data in system memory. This unit only creates write requests. Only a single word (the
status word) is written to the system memory.
Read Data Path Unit (RDP): Reads data from UniPro and writes raw data to system memory. This unit creates only write
transactions. The size of the burst is maximal as long as there is enough data left to transfer.
Write Data Path Unit (WDP): Writes data to UniPro and reads raw data from system memory. This unit creates only read requests.
The size of the burst is maximal as long as there is enough data left to transfer.
While there are several clients which request accesses to the system bus, an arbiter controls the access to the AXI Inititator interface.
While on one hand the arbiter has to provide an optimal occupation of the system bus, the arbiter also has to support special
application requirements, for example, a constant large data stream. To help those requirements, the arbiter implements the following
default arbitration priority scheme (from highest priority to lowest):
1. Fetch task management requests. Controller access: Read data from system memory. Based on the JEDEC specification, task
management request must have the highest priority. As such, those are handled immediately.
2. Read DMA channel. RDP access: Writing data to system memory. This supports applications such as audio or movie playback where
the application interface relies on the data.
3. Fetch new requests and PRDs. Controller access: Read data from system memory. The priority has been increased to get the PRDs
as quick as possible to ensure high data rates at both the directions.
4. Write DMA channel. WDP access: Read data from system memory.
5. Store response data. Controller access: Write data to system memory

Finishing a transfer has the lowest priority. This is based on the assumption that data accesses have a higher priority in order to
maximize the data throughput.
Additional to the simple fixed priority scheme, starvation prevention is added to the arbiter. This is done by automatically aging the
requests which have been made to the arbiter, thus elevating its priority.
Aging is done on a per-request basis, resulting in an aging impulse every time the arbiter makes a selection. Aging is based on a pre-
assigned priority. The starting priority can be assigned to the individual channels through a parameter at design time. Those priorities
are based on the available system performance and latencies introduced by the system architecture. The initial priorities can vary
slightly depending on the system architecture
There are some additional parameters which have to be considered when making priority assignments or analyzing the requests
pattern:
1. RDP and WDP support pipelined transactions. This means as long as there is free space or data in the internal FIFO, those units
create requests. The number of outstanding requests can be configured and is determined by the size of the FIFOs.
2. Because the controller resource is shared by different types of transactions (task start, command header read, PRD fetch), the host
controller internally determines which of the requests has a higher priority. The following scheme is implemented (From highest
priority to lowest):
a. PRD fetch
b. Command header fetch
c. New task fetch

9 Power
9.1 Power Domain
The UFS power domain includes the following elements
1. UFS Subsystem SIP (Synopsys controller and logic, fabric, bridges and glue logic)
2. MIPI MPHY and PGA
A single PGCB block controls the power of the UFS controller and supporting logic. A separate PGCB in PGA is used to control PHY PCS
power delivery.
9.1.1 UFS Subsystem SIP Power Domains
The UFS SS is functionally a Vnn IP but is partitioned into both Vnn (with Vnn_gated domain) and Vnnaon domain. Vnnaon domain is
required for HW autonomous retention. Various parts of UFS SS, UFS Host controller and Unipro will be partitioned between Vnn_gated
and Vnnaon domain as shown in Figure: UFS SS Power Domains. With Vnn gated state (S0-ix), IP doesn’t master any SOC resources like
SB/ Prim fabric. Hence it does not send any Vnn_req/ack to PMC.
1. Vnn Gated (Power Gated): The majority of the logic is located in the Vnn Gated domain and can be powered off by the SoC when not
required (d0i2/d0i3/d3 or UFS Hibernate states). The indication to PMC is in terms of d0i3/ d3 from IP. Note, d0i2 indication and d0i3
indication are OR’ed together into one signal.
2. Vnn Ungated: A small section of logic is located in the Vnn ungated domains. At S0, power is available to this section and it contains
the logic required to control the power gating for the UFS SS (PGCB) and registers for bridge and convergence layer. To reduce
software and hardware complexity the IOSF2AXI Bridge PCI configuration space for UFS and the internal configuration registers are
implemented using retention flops. i.e. the value is stored even though the IOSF2AXI bridge power is disabled when UFS is power
gated.
3. Vnnaon: Vnnaon contains UFS low power FSM and registers that needs to be retained in power gated (Hibernate) state for UFS
controller and unipro.

9.2 Power States
SCS_UFS supports HW controlled D0i2 and SW controlled D0i3/D3 states. UFS controller enters IPAPG state when the controller is in
any one of the supported device states. UFS has a dedicated PCI config space with D3 control bit and a dedicated register within the
IPs MMIO to place it in D0i3. PHY will be put in ultra low power hibernate mode when SW has programmed ULP Hibernate enable
attribute of the PHY (included in sequence).
9.2.0.1 D0i2 - Auto Hibern8 (Hibernate)
UFS supports Auto-Hibern8 flow to allow HW managed power gating based on Auto-Hibernate Idle Timer (AH8IT). The UFS driver uses
Hardware AH8IT (Auto Hibernate Idle Timer) in order to count the idle time of the UFS (the value is set by the host driver). This timer
starts decrementing when the host controller is IDLE and upon timer expiry, the controller places the Unipro link in Hibern8 state. This
timer is reloaded every time the link transitions out of Hibern8 state.
In this mode, UFS is in IP accessible mode. In this mode, though the UFS controller and PHY is power gated, any access will cause a
wakeup event and will be served. From the Host standpoint, the UFS is present and available for access.
9.2.0.2 D0i3/D3 - SW Hibern8 (Hibernate)
SW driver configures UFS controller by writing to Power Control Enable register (PCE) PCE.D3HEN (D3-Hot Enable) or REG_D0I3.D0i3
(D0i3). UFS provides both a D0i3 (UFS_pmc_d0i3_comb) and D3 (UFS_pmc_d3_comb) indication to PMC.
Controller gated domain will be power gated and Unipro, PHY will be in Hibernate mode. SW should write 0x17
(DME_HIBERNATE_ENTER) to controller register UICCMD.CMDOP to enable HIBERNATE entry before setting all the subsystem H8
related registers. PHY can be further put into Ultra low power (Hibernate ULP) to reduce power in this state. SW needs to set RMMI
ulp_h8_en_rw attr (register: CBULPH8) before software can put UFS in Hibernate (H8). When exiting H8 ULP, the ulp_h8_en attr should
be cleared after H8 exit. In this mode, certain M-PHY power rails can be gated for further power saving when entire RMMI link enters
H8.
BIOS must ensure to clear PCE.HAE to 1’b0, set PCE.D3HEN to 1’b1 and set PMECTRLSTATUS.POWERSTATE to 2’b11 for D3 SW H8.
https://docs.intel.com/documents/pch_doc/NVL/PCD-H/HAS/NVL_UFS_SubSystem/NVL_UFS_SubSystem.html 37/77

IP’s D0i3 indication to PMC is implemented as the logical OR of D0i2 and D0i3 device states.
In this mode, IP can be either in IP accessible mode (D3/ D3 Hot) or IP inaccessible mode (D3 cold). In IP inaccessible mode, any
access to the IP will result in an error on the IOSF. No wakeup trigger to the IP shall be issued. Software can’t place the UFS in IP
inaccessible mode, only the PMC can.
IP inaccessible mode is only used during initial bring up, in function disable state and s0ix with VNN removal state.
Auto-Hibern8 allows for more frequent transition of the link into Hibern8 state but does not result in assertion of controller D3 indication
(UFS_pmc_d3_comb) to PMC. To support S0ix with Auto-Hibern8 enabled, the UFS driver is expected to support a Auto-Hibern8 to
D0i3/D3 Entry flow:
1. UFS host driver configures AH8IT to enable auto-hibernate
2. UFS will request for PG upon AH8IT expiry which puts the link in Hibern8 state.
3. At some point, the UFS host driver determines D0i3/D3 entry (based on OS) and writes to the appropriate register bit(s) to enable
D3Hot or D0i3.
a. This action would cause the IP to exit power gating which then causes the controller to re-load AH8IT.
4. Upon AH8IT re-expiry, the link re-enters Hibern8 state which results in the UFS controller re-requesting to be PG.
With the controller now in SW controlled D0i3/D3, exit from this state can only happen via driver interaction:
1. Driver moves controller back to D0
a. This causes the UFS to exit PG
2. UFS-Controller would reload the AH8IT value and PG-entry will re-occur upon timer expiry and link will go in Hibern8 state.

In IP Accessible mode, any access to IOSF primary interface or IOSF sideband will cause a wakeup event to UFS. This will cause UFS to

9.2.1.5 Save and Restore Requirements
In Auto Power gate mode, in S0 state, The UFS controller performs its own SAVE operation for context residing in the IP’s Vnn_gated
domain. This occurs whenever the link is placed in Hibern8 state but SOC is not in S0ix. Upon exit from Hibern8 state, the UFS controller
will perform it’s own RESTORE operation.
When SOC is placed in S0-ix, Vnn is gated off in S0-i2.1 and deeper. SOC is required to save context from above mentioned registers
that are residing in Vnn-ungated domain. Once the SoC exits from low power mode and Vnn is enabled these values can be restored.
MMIO space is not accessible through IOSF-P when device is in D0i3/D3 but is still accessible via IOSF-SB.
During the SAVE operation, PMC asserts pmc_ufs_ip_wake which forces the IP to exit Power gated state. SRM will perform IOSF-SB
reads to save IP content. Upon completion, PMC will deassert pmc_ufs_ip_wake which allows UFS to re-enter Power gated state (since
IP is still in D0i3/D3) once the link re-enters Hibern8 on AH8IT expiry.
When PHY exits from Hibernate Power gated state, UNIPRO restores PHY settings. SOC Save/ Restore does not need to do this.

9.3.2.8 Latching PCE values
The PCE PCI Register is located in the IOSF2AXI Bridge. Hence, when UFS is placed in a power gated state it is moved to a retention
state. I.e. its value is preserved but it does not drive it out.
Various bits in the PCE affect the power gating and wakeup sequence of UFS and hence it is required to latch these values prior to
power gating and use the latched values. (as the unlatched values will be 0 when in power gating)
9.3.2.9 Latching of D3/D0i3 values
During power gated state all the output from UFS are placed in a fixed predefined value by the firewall logic. To provide a correct
D3/D0i3 indication to the PMC, the D3 and D0i3 values are latched in the always on domain (PGCB, CDC. Etc..) when UFS is placed in a
power gated state. This way the value of these bits send to the PMC or any other IP, always reflect the actual value there programed by
software and not the fixed value set by the firewalls.

10.2 Clock Gating
10.2.1 Downstream Access
Downstream access from the host enables the AXI clock requests. Transactions are halted in the IOSF2AXI Bridge until clock available.
The flow is as follows:
1. IOSF Access arrives from PSF to IOSF2AXI Bridge
2. IOSF Primary CDC detects access over IOSF ISM Change and requests IOSF Clock
3. IOSF clock is Enabled
4. Transaction Enters IOSF2AXI Bridge
5. Bridge Indicates IOSF Primary Clock is required and the local gater is opened.
6. Transaction is converted to AXI.
7. IOSF2AXI Bridge indicates AXI Clock is required
8. IOSF2AXI Bridge Gater waits for axi_clk_ack
9. When both ack arrive AXI Clock is enabled to IOSF2AXI Bridge and AXI Transaction moves to AXI Fabric.
10. AXI Fabric cactive Input is asserted, indicating to CDC to keep AXI clock active.
11. Transaction outputs from the AXI Fabric to UFS controller, but is blocked since UFS controller still doesn’t have clocks enabled (Clk_I,
Sysclk).
12. AXI fabric cactive at the UFS is asserted, indicating to the UFS controller to ungate its clocks.
13. Transaction outputs from the AXI fabric to the controller AXI Interface and a response or completion is send up.
14. Once transaction finishes, hysteresis timer starts to work for pre-determined cycles
15. One timer finishes, the AXI clock request is deasserted (assuming no longer required by IPs)
16. CDC disables AXI clock.
10.2.2 Upstream Access
Upstream access refers to the IP reading or placing data in the Host Memory.
For an upstream access to happen the clocks should already be toggling since upstream traffic is in response to a command from host.
As the transfer passes from the IP -> AXI Fabric -> IOSF2AXI Bridge -> PSF, each element shall request the clock from the CDC and
ensure clock availability continuity.
The flow is as follows:
1. IP issues am upstream AXI access
2. AXI access enters AXI fabric
3. AXI fabric asserts the cactive in the Port as indication there is active AXI transaction.
4. Based on the indication the IOSF2AXI bridge gater it opened and AXI clock is provided to the IOSF2AXI Bridge.
5. Bridge deasserts the AXI IDLE indication to indicate clock is required.
6. IOSF2AXI Bridge asserts the IOSF Primary Clk Req to indicate IOSF Clock is required.
7. IOSF Primary Clock CDC requests the IOSF Primary Clock from the trunk.
8. Trunks provides clock and clk_ack indication.
9. IOSF2AXI Bridge now has IOSF Prim clock.
10. AXI Transaction is converted to IOSF and send from Bridge to PSF.
11. Based on type of transaction clocks are kept on or can be disabled.
IF Transaction is Write Posted - Clocks are disabled once transaction clears the IOSF2AXI Fabric

10.3 UFS Power Flows
A: Aon Power Up
1. VnnAON powered.
2. Global Reset de-asserted.
B: Vnn Ungated Power Up
1. VnnUnGated powered up.
2. PGCB Reset de-asserted.
3. Open the firewalls between VnnAon to the VnnUnGated (purple portion).
C: IP inaccessible Exit
1. PMC assert wake to UFS.
2. UFS PGCB deasserts pgcb_pmc_pg_req_b asking PMC to ungate UFS.
3. PMC deasserts pmc_pgcb_pg_ack_b acknowledging ungate request.
4. pmc_pgcb_pg_ack_b signal is used for notifying the UFS IP that power is valid (LP_Pwr_ready).
5. Assertion of original pgcb_isol_en is not used to open UFS FW, it’s done to avoid reset propagation from VNNgated to AON logic (it
used to close the FW, and full control of the other IP FW).
6. Since clocks enable of UFS is function of PGCB FSM restore state it will not toggle during reset de-assertion and propagate to AON.
This behavior is acceptable since AON is at reset state.
7. Force reset to all IP domains propagated with free running PGCB clock.
8. Deassert reset.
9. Delay by 8 clocks “LP_Isol_En” open firewalls between VnnGated and VnnUngated only after reset de-assertion, to avoid reset
propagation from UFS VNN Gated portion to the AON domain.
10. PMC deassert wake and move the UFS to Active mode.
D: IP Accessible entry
1. UFS Auto H8 ILDE Timer count down to 0 UFS move to Auto Hibernate state, AON FSM will save VNNgated content into AON Logic.
2. UFS IP will assert HIBERNATE signal, that fed into the CDC. CDC aggregate all clocks requests and hibernate signal if all are
deasserted it will generate PGCB Power Ready which start IP Accessible entry.
3. PGCB will assert pgcb_isol_en in order to close the firewalls between VnnGated and VnnUngated, to avoid reset propagation from
VNNGated to AON.
4. Force reset to all IP domains excluding AON since FW is closed.
5. pgcb_force_rst_b_delay(LP_reset_en) is not used for FW closing since the FW should be closed prior to reset assertion.
6. PGCB will assert “pgcb_pmc_pg_req_b” notifying the PMC that it should power gate UFS.
7. PMC will assert “pmc_pgcb_pg_ack_b” acknowledging power gating request.
8. UFS power is gated.
E: IP Accessible exit
1. UFS Host driver access UFS, PGCB deassert “pgcb_pmc_pg_req_b”
2. PMC deassert “pmc_pgcb_pg_ack_b” acknowledging power request, this signal is also used to notify the UFS IP that power is resumed
through “LP_Pwr_Ready”
3. Assertion of original pgcb_isol_en is not used to open UFS FW, it’s done to avoid reset propagation from VNNgated to AON logic (it
used to close the FW, and full control of the other IP FW).
4. Since SNPS IP requires all its clocks to exit from power gating, UFS Mux all UFS clocks with PGCB clock to break the coupling between
UFS and MPHY.
5. Force reset to all IP domains propagated with free running PGCB clock.

11.1 VISA

11.2 Security Plugin
Applying security to DFx is critical in protecting internally stored keys and other intellectual property of SOC. However, it is necessary to
enable our customers with tools through third-party vendors and provide on-die DFx features necessary so they can perform debug and

customer to bring the product to market.

11.6 SRAM bypass controls
Refer HSD 15013604201
SoC DFX needs controllability of the SRAM boot related controls. Default drive can be as per functional needs, but expose four new
wires , as below, to SoC to allow SOC to override/control:
input [1:0] dfx_sram_bootload_bypass_mode
input [1:0] dfx_sram_bypass_mode
input dfx_sram_ext_ld_done
input dfx_sram_mode_sel
When dfx_sram_mode_sel is ‘1’, the other 3 signals will follow the value driven from SOC. This would allow DFx to override this interface
early before UFS comes out of reset.

13.6 KAT support
KAT self test is not enabled in NVL.
The UFSHC provides KAT self test feature as defined in the vendor UFSHC databook Table: Reference Documents. KAT enable is
provided as input wire to the UFSHC from fuse proxy. The register to check status are available in the converge layer registers. By
default KAT is not enabled. This has to be enabled (via fuse in fuse proxy) prior to Host controller reset or hibernate exit as the KAT is
triggered on these conditions.

14.1.4.1 Basic PCI Configuration
This document does not cover the recommended programming for conventional PCI and PCI Express registers unless there is a special
requirement specific to the Root Port. It is assumed that the BIOS/software will set these registers up accordingly. System BIOS should
enumerate and configure all root ports as PCI-PCI bridges in compliance with PCI to-PCI Bridge Architecture Specification v1.2. System
BIOS owns the configuration and reporting of the PCI Express* interrupt routing schemes for the root ports and for the secondary buses
behind them correctly.
UFS PCI config and MMIO space, below is the detail to use (Refer PCD IOSF SB excel for port ID):
To do CFG space access, do sideband CFGREAD/CFGWRITE opcode cycles with: > UFS Destination ID FID Address (16bit) = As
mentioned in iosf2axi_pci_configreg
To do MMIO space access, do sideband MREAD/MWRITE opcode cycles with: > UFS Destination ID FID Address (16bit) = Programmed
BAR value in iosf2axi_pci_configreg + the offset of register in scs_dwc_ufshc or scs_converge
BIOS enumerates, as per PCI spec, the UFS host controller once ESE does hand off.
Note: No need to program BAR1 register, as it’s only applicable for ACPI mode, which Is not POR.

14.3.4 UFS Disable
Also see Function Disable.
14.3.4.1 Options available to disable UFS
In all the three options, PMC should NOT see UFS as disabled.
a. If UFS_BIOS_DISABLE (Used for SKU disable)
a. BIOS reads IOSF-SB PGA register (PGA buttress SB PortID, Offset 0x3500, bit 8) to check if UFS is SKU disabled.
b. Follow disable flow (PSF disable etc.)
b. If BIOS static disable (via BIOS menu)
a. BIOS reads the BIOS menu setting to check if UFS is BIOS menu disabled.
b. BIOS does NOT use the BIOS static disable flow with PMC i.e., BIOS does not use the ST_PG_FDIS_PMC register handshake, the
effect is that PMC will release the UFS out of reset.
c. Follow disable flow (PSF disable etc.)
c. If UFS card is not detected (in M.2 form factor, via GPIO)
a. No-Device-Attach scenario
b. Flow is same as BIOS static disable flow
14.3.4.2 BIOS Disable actions
If BIOS detects UFS is disabled via any of the above methods,
a. it will set the calibration DONE flags (to quickly get to ultra low power (ULP) mode),
b. proceeds with the MPHY init sequence and,
c. subsequently follows the UFS disable sequence (psf disable etc.)
Refer Appendix B - BIOS Psuedo Code for pseudo code.

while(rd_data == 0) begin
readDME(32'h00C10005, rd_data);

end
